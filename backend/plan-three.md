# Backend Plan — Step 3: CSV Output for ML Model

## Scope of this step

Extend the existing `POST /api/predict` response to include a CSV string alongside the JSON features.

- **Frontend** continues to read `features.*` as before — nothing breaks.
- **ML teammate** reads the new `csv` field directly from the JSON response, no separate endpoint, no file I/O on either side.

**Still NOT in scope:** the ML model itself, model inference, model output, deployment.

---

## Response shape change

### Before
```json
{
  "status": "ok",
  "features": {
    "datetime": "2026-05-18T14:00:00",
    "waveHeightM": 0.04,
    ...
  }
}
```

### After
```json
{
  "status": "ok",
  "features": {
    "datetime": "2026-05-18T14:00:00",
    "waveHeightM": 0.04,
    ...
  },
  "csv": "datetime,wave_height_m,wave_direction_deg,...\n2026-05-18T14:00:00,0.04,204,...\n"
}
```

The `csv` field is a plain string with a header row and a single data row — already generated by `app/utils/csv_export.py`, now just exposed in the response.

---

## File-by-file changes

### `app/schemas.py`
Add one field to `PredictionResponse`:

```python
class PredictionResponse(CamelModel):
    status: str
    features: FeatureRow
    csv: str
```

Field name stays `csv` over the wire too (single word — `alias_generator=to_camel` leaves it alone).

### `app/routes/prediction.py`
Call the CSV utility after building the feature row and include it in the response:

```python
from datetime import datetime

from fastapi import APIRouter

from app.schemas import PredictionRequest, PredictionResponse
from app.services.aggregator import build_feature_row
from app.utils.csv_export import feature_row_to_csv


router = APIRouter(prefix="/api", tags=["prediction"])


@router.post("/predict", response_model=PredictionResponse)
async def predict(body: PredictionRequest) -> PredictionResponse:
    # Combines the validated date and time into a single datetime for downstream use.
    target = datetime.strptime(f"{body.date} {body.time}", "%Y-%m-%d %H:%M")

    row = await build_feature_row(target)
    csv_output = feature_row_to_csv(row)

    return PredictionResponse(status="ok", features=row, csv=csv_output)
```

No other files change. No new dependencies. No env vars.

## What the ML teammate needs to do

He hits the endpoint normally and extracts the `csv` field. Two common ways:

### Python with pandas (most likely what he wants)

```python
import io
import requests
import pandas as pd

response = requests.post(
    "http://localhost:8000/api/predict",
    json={"date": "2026-05-18", "time": "14:00"},
).json()

# Parse CSV string straight into a DataFrame — no file on disk needed.
df = pd.read_csv(io.StringIO(response["csv"]))

# Feed straight to the model:
prediction = model.predict(df)
```

### Save to file if needed

```python
with open("input.csv", "w") as f:
    f.write(response["csv"])
```

### Test from the command line (no Python needed)

```bash
curl -s -X POST http://localhost:8000/api/predict \
  -H "Content-Type: application/json" \
  -d '{"date":"2026-05-18","time":"14:00"}' \
  | jq -r .csv
```

That prints the CSV directly to the terminal — handy for sanity checks without writing any code.

### What he should expect in the CSV

- Always exactly **two lines**: header + one data row.
- Column count and order match the `FeatureRow` schema fields (16 columns).
- Missing values appear as empty cells (CSV-native nulls) — `pd.read_csv` will turn them into `NaN` automatically.
- Column names are `snake_case` (model-friendly), not `camelCase`.

## Security requirements — non-negotiable

The CSV and the JSON response must contain **only forecast data**. Nothing else. Specifically:

### Must NEVER appear in any response (CSV or JSON):
- `STORMGLASS_API_KEY` or any environment variable value
- Any internal file path (`D:\Projects\...`, `/home/...`, `outputs/...`)
- Server hostname, internal IPs, or port numbers
- Stack traces, exception type names, or raw upstream error messages
- Database connection strings (none exist now, but stating the rule upfront)
- Any field added to `Settings` in `app/config.py` — config stays server-side, period

### How the rules are enforced in code

1. **`csv_export.py` only serializes `FeatureRow` fields.** `FeatureRow` contains exclusively weather/marine/astronomy data. Never add config, credentials, or internal state to this schema.
2. **Service error handlers (`marine.py`, `weather.py`, `tides.py`) catch exceptions, log to `stderr`, and return `None` for their fields.** They never propagate the exception or its message to the client.
3. **No `print(settings.stormglass_api_key)` anywhere.** When debugging API auth, print response status codes, never the request headers.
4. **CORS stays restricted** to the origins listed in `.env` (`http://localhost:5173` etc.). Never `*`.
5. **`.env` is gitignored** (already configured in step 1). Verify with `git status` before pushing — `.env` should never appear.
6. **`.env.example` only contains placeholder values** (`STORMGLASS_API_KEY=your_key_here`) and is the only env file committed.

### Manual security check before considering step 3 done

Run this command and confirm none of the values from `backend/.env` appear in the output:

```bash
curl -s -X POST http://localhost:8000/api/predict \
  -H "Content-Type: application/json" \
  -d '{"date":"2026-05-18","time":"14:00"}'
```

Search the response for your StormGlass key, the literal string `STORMGLASS`, and the substring of any other secret. None should be present.

### What's safe to include (and why)

- Site latitude/longitude (`45.5482, 13.7296`) — the entire app is *about* this fixed public location. Not a secret.
- The site timezone — public information.
- Forecast data values — public data, fetched from public APIs.

## Done when

1. `POST /api/predict` returns a JSON body containing all three keys: `status`, `features`, `csv`.
2. The `csv` field is a non-empty string with one header line and one data line. Newline at the end.
3. Frontend still renders correctly — no broken fields, no parse errors in the browser console.
4. `pd.read_csv(io.StringIO(response["csv"]))` in Python produces a single-row DataFrame with 16 columns.
5. Manual security check above passes — zero secrets in the response.
6. No new env vars, no new dependencies, no new endpoints.

---

## Why Option 3 over Option 1 (file on disk) or Option 2 (CSV-only response)

- **Option 1** would mean the ML teammate has to read the filesystem of whatever machine the backend runs on. Fine locally, painful the moment you deploy.
- **Option 2** would break the frontend the moment the endpoint changes content type.
- **Option 3** is one endpoint, one request, two consumers, zero coordination overhead. The cost is a few extra hundred bytes per response — negligible.
